Developer Guide

How the Registry Works
End to End

This document follows a single registration request from a person clicking Submit all the way to a company becoming a legal entity in the register. Written for new developers joining the team.

Background

Why this system exists

Someone has decided to start a company. Before it legally exists, it must be registered with the national registry — because a company does not exist until the state says it does.

Registration is not something you do on a whim. Before making an application, an identity-verified applicant must gather the necessary information: a proposed company name, at least two directors who have verified-person records, and a registered office address. The registry website sets out the full process clearly, and reading it carefully before starting an application will save time.

This document follows what happens from that moment — a person clicking Submit — all the way to a company becoming a legal entity in the register.

Authentication

The applicant proves who they are

Before the applicant can do anything in the registry, they must authenticate. This is not a simple username and password check. The registry uses a third-party identity broker that is compliant with OIDC — OpenID Connect — an open standard for identity verification built on top of OAuth 2.0.

The applicant's browser is redirected to the identity broker, which coordinates with the appropriate identity provider. The Registry only accepts identity providers that perform identity verification as part of signup and issue a verified natural-person digital identity. Depending on the configuration, the applicant may also be required to complete multi-factor authentication — for example, confirming a code sent to their phone. This entire flow happens between the browser and the broker's own infrastructure — it does not touch the registry server at all.

The Registry consumes verified identities This project does not perform identity proofing itself. It trusts the verified natural-person claim supplied by the identity broker for users, and it trusts verified-person records supplied by external verification services for natural persons named in company data.

A future project may add a parallel workflow that contacts proposed directors, asks them to complete identity verification through a trusted third-party provider, and resumes examination only when confirmation is received. In this contract, the flag or record must already exist before the relevant command can succeed.
The Authorization Code Flow with PKCE and BFF The registry uses a specific authentication flow called the Authorization Code Flow with PKCE (Proof Key for Code Exchange) and the Backend for Frontend (BFF) pattern. This is the most secure approach available for browser-based applications. Authorization Code Flow Rather than sending a token directly to the browser, the identity broker sends a short-lived, single-use authorization code. The registry's backend server exchanges that code for the actual token. The browser never sees the token itself. PKCE — Proof Key for Code Exchange Before the flow starts, the backend generates a secret and sends a one-way hash of it to the broker. When the backend later exchanges the code for a token, it sends the original secret. The broker verifies it matches the hash. Even if an attacker intercepts the authorization code in transit, they cannot exchange it — they do not have the secret. BFF — Backend for Frontend The registry's backend acts as an intermediary for all authentication. The Angular frontend never handles tokens at all. Instead it communicates with the backend using a secure HttpOnly cookie that JavaScript cannot read. Tokens live only on the server. This eliminates an entire class of attacks where malicious JavaScript injected into the page steals tokens.
Redis — where the session lives When authentication completes, the backend receives the access token, refresh token, and the user's identity claims from the broker. These are not stored in Postgres — a database round trip on every request would be too slow. Instead they are stored in Redis, which runs as a pod inside the Kubernetes cluster and can retrieve a record in microseconds.

Before anything is written to Redis, the token data is encrypted. The encryption is handled by OpenBao's Transit engine — a feature OpenBao describes as encryption as a service. OpenBao runs on dedicated VMs outside the Kubernetes cluster, on a private network reachable from the cluster but not from the public internet. The backend sends the plaintext token data to OpenBao Transit over mutual TLS. OpenBao encrypts it using a key that is stored inside OpenBao and never leaves it, then returns the ciphertext. The backend stores that ciphertext in Redis.

The backend generates a random session ID and uses it as the Redis key. Under that key it stores the encrypted payload containing the access token, refresh token, and the applicant's verified identity claims. All encrypted by OpenBao Transit before storage.

The session ID is set as a cookie with three security flags. HttpOnly means JavaScript cannot read it — it is invisible to scripts, which defeats XSS token theft. Secure means the browser will only send it over HTTPS — never over plain HTTP. SameSite=Strict means the browser will not send it on requests originating from other sites — which defeats Cross-Site Request Forgery (CSRF) attacks. All three flags together. JavaScript cannot read it. It is a pointer into Redis — useless without the server, and useless on the server without OpenBao to decrypt the payload.

When the applicant logs out, the backend deletes the Redis entry. The session is immediately and completely invalidated.
Is it not risky to send plaintext tokens to OpenBao over the network? This is a reasonable concern, especially given that OpenBao runs on separate VMs outside the cluster rather than as an in-cluster service. The call does cross a network boundary.

The mitigations are:

All communication between the Kubernetes cluster and the OpenBao VMs uses mutual TLS — both sides present certificates and verify each other. The plaintext is protected in transit. An attacker intercepting the network between the cluster and OpenBao would see only encrypted traffic.

The OpenBao VMs sit on a private network segment with no public internet exposure. Network-level access controls — firewalls and security group rules — permit only the registry backend pods to reach OpenBao's API port. Nothing else can connect.

The alternative — fetching the encryption key from OpenBao and encrypting locally in the application — would expose the key in the application's memory. If an attacker compromised the running process, they could extract the key and decrypt everything in Redis. With OpenBao Transit, the key never leaves OpenBao. The application only ever holds ciphertext. Even a fully compromised backend process cannot extract the key because it does not have it.

Every encrypt and decrypt call is logged by OpenBao's audit system. Any unusual pattern is visible and can trigger an alert.
How OpenBao and Redis are protected How the backend authenticates with OpenBao The backend uses Kubernetes Service Account authentication. Every pod has a short-lived service account token automatically projected into it by Kubernetes. OpenBao is configured to trust the Kubernetes cluster and verifies this token with the Kubernetes API server. The registry backend's service account is permitted to use the Transit engine — and nothing else. No static passwords or API keys are involved. This is the Principle of Least Privilege: every component is granted only the minimum permissions needed to do its job. A compromised backend pod cannot use OpenBao to read other secrets, access other key rings, or perform administrative operations. How static secrets reach the application The Redis connection password and other static credentials are delivered by an OpenBao sidecar agent — a small container that runs alongside the registry backend in the same pod. The sidecar authenticates to OpenBao, fetches the secrets, and writes them to a shared in-memory volume only that pod can see. The registry backend reads them from that file. It never calls OpenBao for these. When a secret rotates in OpenBao, the sidecar picks up the new value automatically. Redis network isolation Redis is a Kubernetes service with no external exposure. Kubernetes Network Policies restrict which pods can connect to it — only the registry backend pods are permitted. TLS encrypts all traffic between the backend and Redis within the cluster. OpenBao VM hardening The OpenBao VMs are not general-purpose servers. They run only OpenBao, are access-controlled, patched, and monitored. OpenBao itself encrypts all data before writing to its own storage — access to the underlying VM disk is not sufficient to read secrets. OpenBao requires an unseal process on startup, meaning the data is protected even if the VM is cloned or its disk imaged. Automatic key rotation OpenBao rotates the Transit engine key on a schedule. Existing ciphertext in Redis can still be decrypted using the previous key version and re-encrypted with the new key without disrupting live sessions. The backend never handles key rotation — OpenBao manages it entirely. The residual risk A realistic attack would require both compromising the private network between the cluster and the OpenBao VMs, and breaking mutual TLS — or alternatively compromising OpenBao itself. Any decrypt operation through OpenBao's API is logged. This is a substantially harder target than reading a key from an environment variable.
Why a broker, not a direct identity provider? The broker decouples the registry from any single identity provider. If the government introduces a new national identity scheme, or an existing provider changes, only the broker needs updating — the registry does not change.

When the registry receives a request, it does not simply accept the session cookie at face value. It actively validates the associated token with the identity broker — confirming the token is genuine, unexpired, and was issued by a trusted broker. Only once that validation passes is the request allowed to proceed.
Trust relationships — how the registry registers with the broker The registry registers itself with the broker as a confidential client. The broker issues a client ID (public) and a client secret (private, known only to the registry backend). The registry uses these when exchanging an authorization code for a token, proving to the broker that it is the legitimate registry.

The broker also registers which redirect URIs are permitted — the exact addresses it may send authorization codes back to. This prevents an attacker registering a fake application and redirecting codes to their own server.

Without this pre-established trust relationship, no authentication flow can take place.
Token lifetimes — the risk and the trade-off Local JWT validation is fast — the registry validates tokens in memory using cached public keys, with no network call. But this introduces a risk worth understanding. Access token — 5 minutes If an access token were stolen — which the BFF pattern makes very unlikely — the attacker has at most 5 minutes before it expires. The registry cannot revoke it sooner because validation is local — a JWT (JSON Web Token) is self-contained and stateless; once issued it is valid until its expiry claim is reached, regardless of what the server does. 5 minutes is the industry standard for high-assurance systems. Refresh token — 1 hour When the access token expires, the backend silently uses the refresh token to obtain a new one — the applicant sees nothing. The refresh token lives only on the server. If a session is forcibly ended, the broker revokes the refresh token immediately and the next renewal attempt fails. JWKS signing keys — cached 10 min, rotated quarterly The broker's public signing keys are cached locally for 10 minutes to 1 hour. The keys themselves rotate roughly four times a year. An emergency rotation is possible if a key is compromised; the short cache duration limits the exposure window. Is this acceptable? For a national companies registry — yes. The BFF pattern keeps tokens off the browser. Access tokens are short-lived. Refresh tokens are server-side and immediately revocable. A higher-assurance system (such as banking transactions) might use token introspection for real-time revocation on every request, accepting the performance cost in exchange.
Accountability and audit Because every request carries a verified identity, we know with certainty who performed every action. Every command, every event in the database, and every audit log entry is tied to a real, verified person. For a national registry this is a legal requirement. If an applicant attempts fraudulent activity, there is an unambiguous, tamper-evident trail.

Network security

Every request to the registry passes through Cloudflare first

Once authenticated, the applicant starts making requests to the registry — creating a draft, submitting an application, and so on. These requests never go directly to the registry server. Before any of them arrive, they pass through Cloudflare.

When the browser makes a request to the registry's domain, it first performs a DNS lookup — translating the domain name into an IP address. That address belongs to Cloudflare, not to the registry server. The registry server itself is never exposed directly to the public internet.

What Cloudflare does Cloudflare operates a global network of data centres that sits between the public internet and the registry server. Every request passes through this network before reaching the registry. WAF — Web Application Firewall Cloudflare runs two rulesets against every request. The first is Cloudflare's own managed ruleset, maintained by their security team and updated continuously as new threats emerge. The second is the OWASP Core Ruleset — an industry-standard set of rules that covers the most common web application attacks: SQL injection, cross-site scripting, malformed headers, and similar exploits. Both run before the request reaches the registry. DDoS protection DDoS protection is automatic on all Cloudflare plans — there is nothing to configure and it cannot be turned off. It operates at three network layers simultaneously: layer 3 and 4 (network-level packet floods) and layer 7 (HTTP-level application attacks). Because Cloudflare's global network handles enormous volumes of traffic across millions of sites, it can absorb and filter attack traffic that would instantly overwhelm a single server. Rate limiting Rate limiting rules restrict how many requests a client can make in a given time window. Importantly, Cloudflare's advanced rate limiting can count requests not just by IP address, but by session token, user ID, or even a field within the JSON request body. For an authenticated API like the registry, this means a compromised account cannot be used to flood the system — the limit applies per verified identity, not just per network address. IP and geographic blocking Requests from IP ranges or geographic regions associated with malicious activity can be blocked outright at Cloudflare's edge — before they consume any registry resources at all. Bot protection Cloudflare assigns a bot score to every request using behavioural analysis, TLS fingerprinting, and machine learning. Known bots and automated scrapers are challenged or blocked. Legitimate automated systems — such as verified search engine crawlers — can be explicitly allowed.
Cloudflare Tunnel — the registry server is never publicly reachable Once Cloudflare decides a request is legitimate, it needs to deliver it to the registry server. This is done via a Cloudflare Tunnel.

A small daemon called cloudflared runs on the registry server. On startup, it establishes a persistent outbound-only connection to Cloudflare's edge network, encrypted with post-quantum encryption. Cloudflare routes approved inbound traffic back through this tunnel to the registry server.

This completely inverts the traditional model. There are no open inbound ports on the registry server. There is no public IP address that an attacker can target directly. The firewall can block all inbound traffic entirely — the tunnel works purely through outbound connections the server itself initiates. Even if an attacker discovered the server's IP address, there is nothing listening to connect to.

This is the Zero Trust security model applied to network access — never expose, never assume trust, always verify. No component is reachable by default. Access must be explicitly granted and authenticated at every boundary.
Defence in Depth Defence in Depth is a security principle: no single control is sufficient, and each layer assumes the others can fail. Cloudflare is the outer perimeter — it handles volume attacks, known exploit patterns, and bot traffic before they reach the system. The registry's own authentication is the inner perimeter — a request that somehow bypassed Cloudflare would still need a valid verified identity to do anything. A stolen session would be limited to a 5-minute access token window. Each layer assumes the others can fail.

Browser → API

The applicant creates a draft

Once authenticated and through Cloudflare, the applicant can begin. The first thing they do is create a draft. A draft is not a registration application. It is a private workspace — a place to gather and organise the required information before making a formal lodgement with the Registry.

The applicant clicks a button — something like Start a new company registration — and the browser sends: 

POST /api/v1/drafts
Every command is idempotent Every command in this system can be safely sent more than once and will produce the same result. This is not optional — it is a core architectural guarantee.

When the browser sends a request, things can go wrong. The network can drop the response after the server has already processed the command. The applicant might click the button twice. A retry mechanism might fire after a timeout. In all these cases, the server might receive the same command more than once.

Without idempotency, a duplicate command could create a duplicate draft, submit an application twice, or worse. For a national registry where legal records depend on correctness, this is unacceptable.

The mechanism is the command ID. Every command carries a unique ID — a UUIDv7 generated by the client before sending. The system records the command ID in a command ledger together with the command outcome. Events caused by that command record the command ID as their causation ID, but causation ID is traceability metadata, not the event's unique identity. Multiple events may share the same causation ID when they were caused by the same command.

If the same command arrives a second time, the command ledger returns the previously recorded outcome and the server performs no side effects again. Nothing is recorded twice. The client cannot tell the difference between a first execution and a repeat.

This also means retrying a failed request is always safe. The client does not need to know whether the first attempt succeeded or not. It simply resends the same command with the same command ID, and the system handles it correctly either way.
Task-based UI vs CRUD Instead of showing a generic form where the user edits fields and clicks Save, the UI exposes named actions — each one corresponding directly to what the user is trying to accomplish. This is a task-based UI, a pattern popularised by Greg Young in the context of CQRS and DDD, and contrasted with CRUD-based UIs where the server must infer intent from which fields changed.

A CRUD-based API exposes resources — you GET a thing, PUT a thing, PATCH a thing. The server has to guess what the user intended by comparing old and new field values. A task-based API exposes intent — the endpoint name tells the server exactly what the user wanted. There is no guessing.

This maps naturally to the backend architecture. Each named action in the UI corresponds directly to a command. Each command is handled by the Decider, which either accepts it and records an event, or rejects it. This is CQRS — Command Query Responsibility Segregation. The user's action, the API endpoint, the backend command, and the recorded event all share the same name and the same intent.
Anatomy of the URL Every part of this URL is a deliberate decision. /api Separates machine-readable API endpoints from browser-rendered pages. /v1 The API version. If the API needs to change in a breaking way, a new version (/v2) is introduced alongside the old one. Existing clients keep working. Critical for a national registry where external systems — tax authorities, banks, legal systems — cannot be forced to update on your schedule. /drafts The aggregate. Each URL segment corresponds directly to a Domain-Driven Design aggregate — the authoritative owner of a concept's state and behaviour. A draft is a distinct aggregate from a registration-application. They have different IDs, different lifecycles, different rules, and different legal meaning. This is the same pattern Stripe uses — payment_intent, customer, invoice are all separate aggregates with their own URLs. No action name on creation POSTing to the aggregate root creates a new instance — the same convention Stripe uses with POST /v1/payment_intents. Action names only appear for operations on an existing resource:

POST /api/v1/drafts/{id}/update
POST /api/v1/drafts/{id}/submit
POST /api/v1/drafts/{id}/cancel

Compare directly with Stripe:

POST /v1/payment_intents/{id}/confirm
POST /v1/payment_intents/{id}/cancel
POST /v1/payment_intents/{id}/capture

The pattern is identical. The URL tells you the aggregate, the instance, and the exact task. A new developer does not need documentation to understand what an endpoint does. The name is the documentation.
What a draft is — and what it is not A draft has no legal existence. It is not visible to the Registry. It cannot be examined or decided upon. It is purely the applicant's private workspace.

A draft tolerates incomplete information. The applicant can create one with minimal details and build it up over several sessions. There is no completeness validation at this stage — that comes at submission.

Several business rules govern drafts: Maximum 5 active drafts per applicant An applicant may not have more than 5 active drafts at any time across all their drafts, regardless of company type. A sixth creation attempt is rejected by the Decider before any event is recorded. Drafts expire after 30 days A draft not submitted within 30 days is automatically expired. A scheduled background job finds active drafts older than the configured expiry threshold and issues an ExpireDraft command to each one through the Decider — the same path as any other command. The state machine is checked, the event is recorded, the audit trail is complete. The business logic is never bypassed. Cancelled and expired drafts are retained Drafts are never deleted. A cancelled draft disappears from the applicant's view but remains in storage permanently. This is a national registry — every action, including abandoned attempts, is retained for audit and legal purposes.

The system creates an empty draft and returns a draft ID — a UUIDv7. Unlike UUIDv4 which is purely random, UUIDv7 embeds a millisecond timestamp in its most significant bits. This means IDs sort chronologically, which improves Postgres index performance and means any ID in a log immediately tells you when it was created. The applicant can now fill in the form across as many sessions as needed, saving progress as they go.

Browser → API

The applicant submits the draft

When the applicant is satisfied the draft is complete, they click Submit. This is the moment the draft becomes a formal registration application — a legal act of lodgement with the Registry. The browser sends: 

POST /api/v1/drafts/{id}/submit
Three distinct domain concepts It is worth being explicit about what these three things are, because they are often confused:

A Draft is a private workspace. No legal existence. Tolerates incomplete information. Owned by the applicant. Has its own ID.

A Registration Application is a formal lodgement. A legal act. Always complete by definition — submission enforces this. Owned by the Registry. Has its own separate ID. Created from a draft but is a different thing.

A Company Registration is a legal fact recorded in the Register. The company exists in law from this moment. Permanent. Has its own ID — the registration number. Created from an approved registration application but is a different thing again.
What happens when the draft is submitted Submission is a validation gate. The Decider checks that the draft contains everything the law requires:

— A proposed company name
— At least two proposed directors
— Every director has a verified-person record
— Every director confirmed as a natural person
— A registered office address

If anything is missing the command is rejected, the draft is unchanged, and no event is recorded. The applicant sees the errors and corrects them.

If all checks pass, the Decider emits a DraftSubmitted event. The draft is now in its terminal state — it cannot be updated, cancelled, or submitted again. It is retained in storage permanently.

A process manager listens for DraftSubmitted events. When it sees one, it issues a CreateRegistrationApplication command, which creates a new aggregate — a registration application — with its own new ID. The content of the draft is copied in. The draft ID is recorded on the application as an audit reference.
Why a process manager rather than a direct call? The process manager is what makes this an Event-driven Architecture (EDA). In EDA, components communicate by producing and consuming events rather than calling each other directly. The Draft aggregate does not know about the Registration Application aggregate. It simply records what happened. The process manager, listening to the event stream, decides what to do next.

This means the two aggregates are fully decoupled. Adding a new reaction to a DraftSubmitted event — a confirmation email, a supervisor notification — requires no changes to the Draft aggregate at all.

The process manager is idempotent. If it sees the same DraftSubmitted event twice — which can happen in distributed systems — it does nothing the second time. It records the event it has handled and issues commands with stable command IDs so retries return the previous outcome rather than appending duplicate events.
The process manager is a FSM — and must save state durably A process manager is not just a listener that fires and forgets. For any workflow spanning more than one step, it is a finite state machine — it moves through defined states in response to events it receives, and issues commands as a result of those transitions.

The draft submission process is simple — one trigger, one command, one confirmation:

[Waiting]
  → DraftSubmitted received
    → issues CreateRegistrationApplication
[AwaitingConfirmation]
  → RegistrationApplicationCreated received
[Completed]
But longer workflows — approval involving address validation, legal review, or payment — have many states, branches, timeouts, and compensating actions for failures. The FSM must be explicit about all. Why state persistence is critical A process manager that does not durably save its state is dangerous. If it crashes between receiving an event and issuing the resulting command — or between issuing a command and receiving confirmation — it must recover to exactly the right position in the FSM on restart. It cannot guess where it was. It cannot start from the beginning.

Its state must be persisted durably before any command is issued, recovered correctly on restart, and idempotent — replaying the same event must not advance the FSM twice. Implemented as an event-sourced FSM In this system the process manager is implemented using the same pattern as aggregates — event sourcing. Its state transitions are recorded as events in their own dedicated event stream. On restart it replays its own events to rebuild its current FSM position, exactly as aggregates do.

draft-submission-process/{id} event stream:
  DraftSubmissionStarted → state: AwaitingAppCreation
  AppCreationCommandIssued → state: AwaitingConfirmation
  AppCreationConfirmed → state: Completed
This gives a full audit trail of the coordination itself — not just the domain facts. You can answer: what was the process manager doing at 14:32:07? Had it issued the command? Was it still waiting? For a national registry where auditability is a legal requirement this matters. The process manager is not a domain aggregate It does not enforce business invariants. It does not own domain concepts. But it uses the same infrastructure — event sourcing, FSM, event store — for coordination purposes. It is event-sourced coordination infrastructure, not a domain model.
A more complex example — the approval process manager The registration application approval workflow illustrates why a robust FSM is necessary. The approval Decider records the Registrar's decision. The process manager then performs the follow-on registration work:

— Observe RegistrationApplicationApproved
— Issue CreateRegisteredCompany with a new Registration Number
— Wait for RegisteredCompanyCreated
— Notify the applicant only after the registered company exists

The approval process manager FSM:

[Waiting]
  → RegistrationApplicationApproved → [CreatingRegisteredCompany]
[CreatingRegisteredCompany]
  → RegisteredCompanyCreated → [NotifyingApplicant]
  → Failed → [CompensatingAction]
[NotifyingApplicant]
  → NotificationSent → [Completed]
Each state transition is an event in the process manager's own stream. If it crashes while in NotifyingApplicant, it restarts in that state — it does not re-create the registered company or re-approve the application. It simply retries the notification. No domain state is corrupted. No business rules run twice. Compensating actions If registered-company creation fails after approval is recorded, the process manager moves to CompensatingAction and issues a corrective command — recording the failure, alerting operations, ensuring no inconsistent state. This is the Saga pattern (described in detail in Chris Richardson's Microservices Patterns) — a sequence of local transactions where each step emits an event triggering the next, and failures trigger explicit repair actions. In this registry, compensation does not alter or delete legal facts that have already been recorded. It records the failure, alerts operations, and allows the missing registration step to be retried. The event-sourced FSM makes each step explicit, auditable, and recoverable. Timeouts If registered-company creation or applicant notification has not completed within a configured threshold, the process manager records a timeout and retries or alerts operations. The timeout is tracked against the process manager's own event stream timestamp — no separate timer infrastructure needed.

Inside the server

The server processes the submit command

The request has passed Cloudflare, been authenticated, and arrived at the registry's Clojure backend. Now everything happens in memory — fast, pure, and auditable. Let us follow it step by step.

1. The middleware stack Before the request reaches any business logic, it passes through a stack of middleware functions. Each one does one specific job. Token validation The session cookie is read. The backend looks up the session ID in Redis, retrieves the encrypted payload, calls OpenBao's Transit engine to decrypt it, and validates the access token using the cached JWKS public key. If anything fails — missing cookie, expired token, invalid signature — a 401 is returned immediately and nothing else happens. Correlation ID The middleware reads the x-correlation-id header from the request. If one is present it is used. If not, a new UUIDv7 is generated. It is attached to the request and will flow through every subsequent operation — every log line, every event, every database write — for the lifetime of this request. JSON deserialisation The request body is parsed from JSON into a Clojure map.
2. The router Reitit — the routing library — matches the URL path to the correct handler function. POST /api/v1/drafts/{id}/submit maps to the submit-draft handler. The draft ID is extracted from the path.
3. The handler The handler is deliberately thin. It does four things only:

— Extracts the draft ID from the path
— Extracts the applicant identity from the verified session
— Builds a command map:

{:command/type :SubmitDraft
 :draft-id "01966b3e-..."
 :applicant-id "a7f21c..."} — Passes it to the runner

No business logic lives in the handler. It knows nothing about state machines, business rules, or events. It is purely a translation layer between HTTP and commands.

This separation — handler, runner, Decider — reflects the Hexagonal Architecture pattern (also called Ports and Adapters, described by Alistair Cockburn). The Decider is the pure domain core. The runner is the application layer. The handler is the adapter that connects HTTP to the application. Each layer depends only on the layer inside it, never the other way around.
4. The runner enriches the command Before the command reaches the Decider, the runner adds two fields:

{:command/type :SubmitDraft
 :command/id "01966b3f-..." ← new UUIDv7
 :command/correlation-id "x-corr-id..." ← from request
 :draft-id "01966b3e-..."
 :applicant-id "a7f21c..."} The command/id is the idempotency key. It will become the causation-id on every event this command produces. The correlation-id flows from the original HTTP request all the way into the event store.
5. Loading the aggregate from the event store The runner loads all past events for this draft from Postgres:

SELECT event_data FROM draft_events
WHERE aggregate_id = ?
ORDER BY sequence_number ASC It then folds those events through the Decider's evolve function, one by one, to rebuild the current state of the draft entirely in memory. Folding is a functional programming concept: starting from an empty initial state, each event is applied to the current state to produce the next state. The evolve function takes the current state and one event, and returns the new state. After all events are folded, the result is the current state of the aggregate — complete, correct, and rebuilt entirely from its history. There is no draft table with a current state column. The state is always derived by replaying history. This is Event Sourcing — the event store is an append-only log. Events are written but never updated or deleted. Current state is always a derived view, rebuilt on demand by folding events. This pattern was described by Martin Fowler and Greg Young and is the foundation of the system's auditability guarantees.

For a draft that has been created and updated a few times, this might be three or four events. Replaying them takes microseconds.
6. The Decider The runner passes the rebuilt state and the command to the Decider's decide function. This is where all business logic lives. It runs in two layers, in strict order: State machine check — first Is this command legal in the current state? A draft in Active state can accept SubmitDraft. A draft in Cancelled or Expired state cannot. If the state machine rejects the command, the Decider returns an error immediately. No business rules are checked. Nothing is saved. Business rules — second If the state machine allows it, the business rules run. For SubmitDraft this means checking that all required fields are present and valid — company name, at least two directors, every director has a verified-person record and is a natural person, the applicant has a verified natural-person identity, and a registered office address is present. Each failing check returns a specific error code. Events — if all checks pass The Decider returns a list of events to record. For a successful submission this is a single event:

{:event/id "01966b40-..."
 :event/type :DraftSubmitted
 :event/causation-id "01966b3f-..." ← the command/id
 :event/correlation-id "x-corr-id..."
 :draft-id "01966b3e-..."
 :applicant-id "a7f21c..."
 :company-name "Acme Ltd"
 :directors [...]
 :registered-office {...}
 :occurred-at "2026-05-26T..."} The Decider is a pure function. It has no side effects. It does not touch the database, the message broker, or any infrastructure. Given the same state and command it will always return the same events. This makes it trivial to test and reason about.

This pattern — a pure function that takes state and a command and returns events — is known as the Functional Event Sourcing Decider pattern, described by Jérémie Chassaing. It is the foundation of how all three aggregates in this system work.
7. Saving events to Postgres The runner takes the events returned by the Decider and writes them to the draft_events table inside a single database transaction:

— The event is assigned the next sequence number for this aggregate
— The command ledger prevents duplicate command handling
— The event records the command ID as causation_id for traceability
— published = false — the outbox pattern flags it for delivery
— If two concurrent requests try to write the same sequence number, Postgres rejects the second — this is the Optimistic Concurrency Control pattern: no locks are held, but conflicts are detected and rejected at write time

If this transaction succeeds, the events and command outcome are durably recorded. If it fails for any reason — constraint violation, database error, server crash — the events are not saved and the command can be safely retried.

The handler returns 200 OK to the browser.
8. The outbox publisher delivers the event The event is now in Postgres with published = false. A background thread — the outbox publisher — polls for unpublished events every 200 milliseconds. When it finds the DraftSubmitted event, it publishes it to RabbitMQ and marks it published = true.

This solves the dual-write problem using the Transactional Outbox pattern. The event is saved to Postgres and published to RabbitMQ as two separate operations — but if the publisher crashes after saving and before publishing, it simply retries on the next poll cycle. Because commands are idempotent and events carry stable event IDs plus causation/correlation metadata, consumers can handle duplicates safely.

The process manager, listening on RabbitMQ, sees the DraftSubmitted event and issues a CreateRegistrationApplication command. A new aggregate is born — the registration application — with its own ID, its own event stream, and its own lifecycle. That is step 06.
Domain events vs integration events — what RabbitMQ actually carries There is an important distinction worth understanding here, because the two terms are often confused. Domain events A domain event is a fact that happened inside a bounded context, recorded by an aggregate. DraftSubmitted, RegistrationApplicationCreated, RegisteredCompanyCreated. These are stored in Postgres in the event store. They are the source of truth. They are immutable. They exist to record the legal and operational history of the system. They are what rebuilds aggregate state when you replay events. Integration events An integration event is a notification published across Bounded Context boundaries — or to external systems — to tell them something happened. A Bounded Context (a DDD term coined by Eric Evans) is a clearly defined part of the system with its own language, model, and ownership. In this project the registry is the bounded context; Draft, RegistrationApplication, and RegisteredCompany are aggregate boundaries inside it. Integration events travel on RabbitMQ and are derived from domain events. It is consumed by the process manager, the projectors, and any future external system that needs to react to registry activity. The relationship in this system The domain event is stored first — durably, in Postgres, in the transaction that validates and saves the command result. Only then does the outbox publisher promote it to an integration event and publish it to RabbitMQ. This ordering is deliberate and critical:

Domain event → Postgres (source of truth)
                 ↓ outbox publisher
Integration event → RabbitMQ (transport)
RabbitMQ is the transport layer, not the source of truth. If RabbitMQ were lost entirely, nothing would be lost from the domain record — all events are safe in Postgres. The outbox publisher would simply re-publish them once RabbitMQ recovered.

In this system the domain event and the integration event carry the same data — they are not different shapes. But they serve different purposes and live in different places. A future external system — a tax authority integration, a public API, an analytics pipeline — consumes integration events from RabbitMQ. It never touches the Postgres event store directly. The event store is internal. RabbitMQ is the public boundary.
At-least-once delivery — what it means and what it guarantees RabbitMQ with a durable resilient cluster and persistent messages provides at-least-once delivery. This is a precise term worth understanding exactly.

What it guarantees: every message that is accepted by the broker will be delivered to a consumer at least once. If RabbitMQ goes down before a consumer reads a message, the message survives in the durable queue and is delivered when the broker recovers. A message is never silently lost.

What it does not guarantee: a message will be delivered exactly once. RabbitMQ re-delivers any message that was not acknowledged by its consumer. If a consumer reads a message, begins processing it, and then crashes before sending an acknowledgement, RabbitMQ will re-deliver that message to the next available consumer. The message will be processed again.

This is correct and expected behaviour — not a bug. It means every consumer in this system must implement the Idempotent Consumer pattern: processing the same message twice must produce the same result as processing it once. This is a well-established pattern described by Gregor Hohpe and Bobby Woolf in Enterprise Integration Patterns.
Do we need an inbox pattern? An inbox is the consumer-side equivalent of the outbox. Before processing a message, the consumer writes it to a local inbox table. If it crashes mid-processing, it recovers from its own inbox rather than waiting for RabbitMQ to re-deliver.

With a durable RabbitMQ cluster and idempotent consumers this system does not need a full inbox pattern. Instead it uses a lighter approach that gives equivalent guarantees at lower complexity. Process manager and projectors These write to the event store or to read model tables. Commands that append events are protected by the command ledger, and projectors are protected by their own projection keys. If a CreateRegistrationApplication command is issued twice with the same command ID, the command ledger returns the previously recorded outcome and no second write is attempted. If a projector tries to write the same row to the examiner work queue twice, the primary key constraint rejects it. No event-store-wide constraint on causation metadata is needed. External side effects — email, external APIs These require a different approach because there is no database constraint to catch duplicates. For each such consumer, a lightweight processed_messages table is maintained, keyed on the message or event ID being consumed. Before processing, the consumer checks this table. If the message ID is already there, it acknowledges the message and stops — the work was already done. If not, it processes the message, records the message ID, and acknowledges. This is the inbox pattern in its minimal form.
The guarantees this approach gives us — precisely stated No domain event is ever lost Domain events are written to Postgres before anything else happens. If the system crashes at any point after that write, the event is safe. The outbox publisher will deliver it when the system recovers. This is an absolute guarantee — not probabilistic. No integration event is silently dropped RabbitMQ's durable cluster ensures every accepted message survives broker restarts. The outbox pattern ensures every domain event is eventually published. Together they guarantee that every domain event becomes an integration event — eventually. Delivery may be delayed by an outage, but it will happen. Every consumer is safe to retry Because every consumer checks message or event IDs before acting, re-delivery of a message produces no additional side effects. A crashed and restarted consumer picks up where it left off safely. What this approach does not guarantee It does not guarantee real-time delivery. If RabbitMQ is down for an hour, integration events queue up in the outbox and are delivered when it recovers. Consumers dependent on those events — email notifications, the examiner work queue — will be delayed by the same duration. For a national registry processing company registrations this is an acceptable trade-off. For a real-time payment system it might not be. The term to remember This architecture provides at-least-once delivery with idempotent consumers. This is the standard, proven pattern for reliable event-driven systems. It is what most serious event-driven architectures use in production. Exactly-once delivery — the alternative — is significantly harder to achieve and comes with substantial performance and complexity costs that are rarely justified.

Inside the server — event-driven

The process manager creates the registration application

The DraftSubmitted event has been published by the outbox publisher. The process manager — a component that listens to the event stream — picks it up and responds. This is the event-driven part of the architecture in action.

What a process manager is A process manager listens for specific events and issues commands in response. It coordinates work that spans more than one aggregate without either aggregate knowing about the other.

In this case the process manager listens for DraftSubmitted and responds by issuing a CreateRegistrationApplication command. The Draft aggregate does not know this happens. The Registration Application aggregate does not know where it came from. The process manager is the only thing that knows about both.

This decoupling is intentional. It means either aggregate can change independently. New reactions to DraftSubmitted — a confirmation email, a supervisor alert, a statistics counter — can be added by writing a new listener, with no changes to any existing code.
What the process manager does When the process manager sees a DraftSubmitted event it: 1. Checks for idempotency Before doing anything, it checks whether it has already processed this event. It does this by querying whether a registration application already exists with this draft ID as its source. If one exists, it stops — this event has already been handled. This is the process manager's own idempotency guard, separate from the command ledger. 2. Generates a new aggregate ID The registration application gets a brand new UUIDv7 — completely separate from the draft ID. These are two different aggregates with two different identities. 3. Issues the command The process manager issues a CreateRegistrationApplication command to the runner, carrying the new aggregate ID, the draft ID as an audit reference, and the full content copied from the draft:

{:command/type :CreateRegistrationApplication
 :command/id "01966b41-..." ← new UUIDv7
 :command/correlation-id "x-corr-id..." ← same as the original request
 :application-id "01966b42-..." ← new aggregate ID
 :source-draft-id "01966b3e-..." ← audit trail back to draft
 :company-name "Acme Ltd"
 :proposed-directors [...]
 :registered-office {...}} Notice the correlation ID is the same one that arrived in the original HTTP request. It flows through the draft command, into the draft event, into the process manager, and now into this new command. Every operation triggered by that original browser request shares the same correlation ID. 4. The runner processes the command The runner follows exactly the same path as step 05 — enrich the command, load events (none yet, this is a new aggregate), call the Decider, save events to Postgres. The Registration Application Decider accepts the command, since a new aggregate starts in its initial state where create-registration-application is the only permitted command. 5. The event is saved A RegistrationApplicationCreated event is written to the registration_application_events table with published = false. The outbox publisher will pick it up and publish it. The examiner work queue projector, listening for this event, will write a row to the examiner_work_queue read model — making the application visible to examiners.
The correlation ID across the entire journey so far From a single click of Submit in the browser, the following has happened:

Browser sends request with x-correlation-id: abc-123
→ DraftSubmitted event stored with correlation_id: abc-123
→ Process manager issues command with correlation-id: abc-123
→ RegistrationApplicationCreated event stored with correlation_id: abc-123
→ 200 OK returned to browser with x-correlation-id: abc-123

If anything goes wrong at any point — hours or days later — a developer can query the event store for correlation_id = abc-123 and see the complete causal chain of everything that happened as a result of that one request. This is Distributed Tracing built into the data model itself — the same concept used by tools like OpenTelemetry and Jaeger, but implemented here through the event store rather than a separate tracing infrastructure.
What the applicant sees From the applicant's point of view, clicking Submit produces one of two outcomes:

Success — the browser receives 200 OK and the UI shows a confirmation: their application has been lodged and is awaiting examination. The applicant is given their registration application reference number.

Validation failure — the browser receives 422 Unprocessable Entity with a specific error code explaining what is missing. The draft is unchanged. The applicant corrects the issue and tries again.

The applicant does not see — and does not need to see — that a process manager created a second aggregate behind the scenes. From their perspective, they submitted their details and received a reference number. The architecture is invisible.

Examiner → API

The examiner picks up the application

The registration application now exists. The outbox publisher has published the RegistrationApplicationCreated event to RabbitMQ. The examiner work queue projector — a consumer listening on that event — has written a row to the examiner_work_queue read model in Postgres. The application is now visible to examiners.

The read model — CQRS in practice This is CQRS made concrete. The write side is the event store — registration_application_events in Postgres. The read side is the examiner work queue table — a denormalised, query-optimised projection of the same data.

The examiner does not query the event store to see applications. They query the read model. It is deliberately denormalised — data that is split across multiple event streams is combined into a single flat table optimised for querying. In a normalised database you avoid repeating data to save space. In a read model you repeat data freely to make queries fast and simple. The examiner's query is a single table scan, not a join across event streams. The two sides have different jobs and different shapes. This is why they are separated.

The read model is not the source of truth. If it were lost entirely it could be rebuilt by replaying all RegistrationApplicationCreated events from the event store through the projector. This is one of the most powerful properties of event sourcing — read models are disposable and rebuildable.

The component that listens to events and writes to a read model is called an Event Projector (or simply a projector). It is a pure consumer — it never issues commands, never modifies the event store. Its only job is to maintain a read-optimised view of the domain. A system can have many projectors for the same event stream, each building a different read model for a different purpose.
The examiner's work queue An examiner logs into the registry system. They see a list of submitted applications waiting for examination, oldest first. They select one. The browser sends: 
POST /api/v1/registration-applications/{id}/begin-examination
What begin-examination does The command flows through exactly the same path as every other command — middleware, router, handler, runner, Decider.

The Registration Application Decider checks the state machine: is begin-examination permitted in the Submitted state? Yes. No business rules apply — any examiner can pick up any submitted application. The Decider emits:

{:event/type :ExaminationStarted
 :application-id "01966b42-..."
 :examiner-id "examiner-007"
 :event/causation-id "..."
 :event/correlation-id "..."
 :occurred-at "2026-05-26T..."} The event is saved. The application is now in UnderExamination state. The examiner work queue projector updates the read model row — the application moves from waiting to in-progress and is no longer visible to other examiners. It is now this examiner's responsibility.
What the examiner does The examiner reviews the application details against the Companies Registration Act. They check the proposed company name, the proposed directors, their verified-person records, and the registered office address. They may consult external sources — the national address database, the existing register — to verify that the address is valid and the name is unique.

This examination is a human process. The system does not automate the decision. The Registrar makes the final call. The examiner's role is to prepare the case for that decision — checking everything the law requires before it reaches the Registrar.

When examination is complete, the examiner escalates to the Registrar. The Registrar will either approve or reject. That is step 08.
Authorisation — RBAC at the handler level The system enforces Role-Based Access Control (RBAC) at the handler level using Claims-based Identity. The identity token from the OIDC broker carries the user's roles, permissions, and verified natural-person identity claim as claims — key-value pairs embedded in the signed token that assert facts about the user. Because the token is cryptographically signed by the identity broker, claims cannot be forged by a client. The middleware extracts these from the decrypted Redis session and attaches them to the request.

An applicant cannot call BeginExamination. An examiner cannot call ApproveApplication — only a Registrar can. A Registrar cannot create or submit drafts on behalf of an applicant.

These checks happen in the handler — before the command reaches the runner or the Decider. The Decider itself does not check roles. Its job is business rules and state machine logic. Role enforcement is the handler's job. The two concerns are deliberately separated — this is the Separation of Concerns principle applied architecturally. Security policy and domain logic can change independently without affecting each other.

Registrar → API

The Registrar approves the application

The examiner has reviewed the application and is satisfied it complies with the Act. The case is escalated to the Registrar. The Registrar reviews the application and clicks Approve Application. This is a decision about the application — not yet the act of registration. The browser sends: 

POST /api/v1/registration-applications/{id}/approve
The approval command The command carries the Registrar's identity from the verified session, never from the request body. The handler injects two functions required for the approval business rules:

{:command/type :ApproveApplication
 :registrar-id "registrar-001"
 :company-name "Acme Ltd"
 :proposed-directors [...]
 :registered-office {...}
 :name-available-fn (fn [name] ...) ← checks the authoritative Register write path
 :address-valid-fn (fn [addr] ...) ← calls address service} These injected functions allow the Decider to check BR-AP-007, BR-AP-009, and BR-AP-010 while remaining a pure function. The Decider does not know how they are implemented, but the name check must use authoritative write-side Register facts, not a stale read model. The name check is deliberately non-reserving in this contract: it does not lock the proposed name.
The Decider runs its most important checks State machine check: is ApproveApplication permitted in ReadyForDecision? Yes. Then business rules:

BR-AP-001 — company name stated
BR-AP-002 — at least two proposed directors
BR-AP-003 — proposed directors are distinct persons
BR-AP-004 — registered office address stated
BR-AP-005 — all proposed directors are natural persons
BR-AP-006 — all proposed directors have verified-person records
BR-AP-007 — company name does not already exist in the Register
BR-AP-008 — company name is not prohibited
BR-AP-009 — registered office address is plausible and valid
BR-AP-010 — company name is not a protected former name

BR-AP-001 through BR-AP-006 repeat submission checks because the Registrar's decision must be made against the authoritative record. Identity verification records do not expire in this contract; revocation is out of scope.

If any check fails the command is rejected. The application stays in ReadyForDecision. Nothing is recorded.
One event — the decision fact If all checks pass, the Decider emits exactly one event:

RegistrationApplicationApproved
 ← the decision fact: the Registrar decided this application complies
 :application-id "01966b42-..."
 :registrar-id "registrar-001"
 :event/causation-id "..."
 :occurred-at "2026-05-26T14:47:23Z" The application transitions to Approved — a terminal state. The application lifecycle is complete. The RegistrationApplication aggregate will never receive another command.

Notice what this event does not contain: a registration number, a company ID, any assertion that the company now exists. That is deliberate. This event records a decision about an application. It says nothing about a company. The company does not yet exist in the system at this moment.

The question of why — when CRA §14 directs registration upon approval and §15 creates legal existence on assignment of the Registration Number — is answered in the next step.
Why not emit RegisteredCompanyCreated here? This is a deliberate architectural decision worth understanding carefully. There are two options:

Option A (chosen): The RegistrationApplication Decider emits only RegistrationApplicationApproved. A process manager reacts to this event and creates a new RegisteredCompany aggregate, which emits RegisteredCompanyCreated on its own event stream.

Option B (rejected): The RegistrationApplication Decider emits both RegistrationApplicationApproved and RegisteredCompanyCreated atomically.

Option B was rejected for three reasons: 1. Aggregates should own only their own facts A RegisteredCompanyCreated event is a fact about a company — not a fact about an application. Placing it on the application's event stream conflates two different domain concepts. The RegistrationApplication aggregate's job ends with the decision. The company's existence is a separate concern that belongs on a separate aggregate. 2. A registered company has its own ongoing lifecycle After registration, a company can change directors, change its registered office, file annual returns, and eventually be struck off or dissolved. These commands and events belong on a RegisteredCompany aggregate with its own state machine. If the company's existence were recorded on the application's event stream, you would have to keep adding commands to a terminal aggregate — one that is supposed to be finished. 3. The identifier changes A RegistrationApplication is identified by a UUIDv7. A RegisteredCompany is identified by its registration number — the permanent legal identity of the company. These are different things with different identifiers. They should be different aggregates.

Process Manager → RegisteredCompany aggregate

The process manager creates the registered company

The outbox publisher picks up the RegistrationApplicationApproved event and publishes it to RabbitMQ as an integration event. The approval process manager — which has been coordinating this workflow — receives it.

What the process manager does The process manager transitions from AwaitingApprovalConfirmation to CreatingRegisteredCompany and issues a CreateRegisteredCompany command to the runner:

{:command/type :CreateRegisteredCompany
 :command/id "01966b51-..." ← new UUIDv7
 :command/correlation-id "x-corr-id..." ← same original correlation ID
 :registration-number "IE-2026-01966b50" ← generated here
 :source-application-id "01966b42-..." ← audit trail
 :company-name "Acme Ltd"
 :proposed-directors [...]
 :registered-office {...}
 :approved-at "2026-05-26T14:47:23Z"} The registration number is generated by the process manager at this point — not by the application Decider. It is the permanent legal identifier of the company. It must be unique across the entire register and must never be reused.

The command is still subject to the final write-side Register guard: the registered company cannot be created unless the company name remains unique against current and protected former names in the authoritative Register. This is a final legal invariant check, not a read-model query. Because name reservation is parked for future legal/product design, this final check is the first point at which the system can make the name legally unavailable to every other registration.
Parked concern — name reservation and locking A fuller registry would likely check availability during draft preparation and lock the proposed name when the draft is successfully submitted. That lock would then be consumed by registration, or released if the application is withdrawn or rejected.

This is deliberately parked in this contract because it needs explicit legal footing in the Act and a careful product model. The current workflow performs non-reserving checks at approval and a final write-side guard before RegisteredCompanyCreated. If that final guard fails, no Registration Number is assigned, no company exists, no applicant registration notification is sent, and the approval process manager records an operational exception for manual resolution.
The RegisteredCompany aggregate is created The RegisteredCompany Decider accepts the CreateRegisteredCompany command — a new aggregate in its initial state accepts this as its first command, after the final Register name check has passed — and emits:

RegisteredCompanyCreated
 ← the existence fact: this company legally exists
 :registration-number "IE-2026-01966b50" ← the aggregate ID
 :source-application-id "01966b42-..."
 :company-name "Acme Ltd"
 :directors [...]
 :registered-office {...}
 :event/causation-id "..."
 :event/correlation-id "x-corr-id..." ← still the original
 :occurred-at "2026-05-26T14:47:23Z" This is the most important event in the system. From this moment, the company legally exists. The registration number is its permanent identity — it will never be reused even if the company is later struck off or dissolved.

The RegisteredCompany aggregate now has its own event stream, its own state machine, and its own lifecycle. Future commands — changing directors, changing the registered office, filing annual returns — will be issued against this aggregate using the registration number as the identifier:

POST /api/v1/registered-companies/{reg-number}/change-director
POST /api/v1/registered-companies/{reg-number}/change-registered-office
POST /api/v1/registered-companies/{reg-number}/file-annual-return
POST /api/v1/registered-companies/{reg-number}/strike-off
The event store is the Register — the read model is derived The authoritative Register is not a database table. It is the event store itself — specifically, the set of registered_company_events streams that contain a RegisteredCompanyCreated event.

If you want to know whether a company legally exists, the ground truth is the event store. If RegisteredCompanyCreated exists in that company's event stream, the company exists.

The register table in Postgres is a read model projection — derived data, shaped for querying. The Register projector consumes registered-company events such as RegisteredCompanyCreated, DirectorChanged, RegisteredOfficeChanged, CompanyStruckOff, and CompanyDissolved, then writes the current query shape to this table. It is queryable, searchable, and fast. But it is not the source of truth.

If the register table were destroyed, no company would cease to exist. The Register projector could replay every registered-company events from the event store and rebuild it completely. The read model is disposable. The event store is not.

The register table must never be written to directly. Only the Register projector may write to it, only in response to registered-company events. Any direct write would create a divergence between the read model and the source of truth.
The complete audit trail — from draft to registered company The full chain now exists across three aggregate event streams:

draft_events/{draft-id}:
  DraftCreated
  DraftUpdated (×n)
  DraftSubmitted

registration_application_events/{application-id}:
  RegistrationApplicationCreated ← source-draft-id links back
  ExaminationStarted
  RegistrationApplicationApproved ← decision fact, terminal

registered_company_events/{registration-number}:
  RegisteredCompanyCreated ← existence fact, The Register
   (future events...)
 Each event stream is owned by its aggregate. Each aggregate records only its own facts. The source-application-id on RegisteredCompanyCreated links back to the application. The draft ID on the application links back to the draft. The full legal history of how this company came to exist is traceable across all three streams by correlation ID and causation ID.

This is not just good engineering. For a national registry, it is a legal requirement.

Summary

Key principles for new developers

You have followed a single registration from a person clicking a button all the way to a company entering the Register. Here are the principles that shaped every decision along the way.

1. The law is the specification Every state, every transition, every business rule traces back to a section of the Companies Registration Act. Before writing any code, the legal instrument was read, the state machine was drawn, and a lawyer was asked to confirm it. The code enforces the law — not a developer's interpretation of it.

When the law changes, the change starts at the state machine, not in the code. The state machine is the specification. The code is the implementation of the specification.
2. Aggregates own their own facts There are three aggregates in the registration domain: Draft, RegistrationApplication, and RegisteredCompany. Each owns exactly the events that happened to it. None of them knows about the others directly.

A Draft does not know that submitting it creates a RegistrationApplication. A RegistrationApplication does not know that approving it creates a RegisteredCompany. This decoupling is intentional. Adding a new reaction to any event requires no changes to the aggregate that emitted it.
3. The event store is the source of truth Events are facts. Once recorded, they are never changed or deleted. The current state of any aggregate is always derived by replaying its event stream — not by reading a state column from a database table.

The Register itself is the set of registered_company_events streams containing RegisteredCompanyCreated. Every read model — the examiner work queue, the register table, the applicant's draft list — is a disposable projection derived from the event store. If any read model were destroyed it could be rebuilt completely by replaying events. The event store cannot be rebuilt from the read models.
4. Commands are decisions, events are facts A command is a request. It can be rejected. It produces no permanent record if it fails. An event is a fact that has already happened. It is permanent, immutable, and represents ground truth.

The Decider enforces this boundary. It takes a command and current state, runs the state machine and business rules, and returns either events or an error. It never touches storage. It never has side effects. Given the same state and command it always returns the same result. This makes it trivially testable and completely auditable.
5. Every command is idempotent Every command carries a unique ID. The command ledger records that ID and the command outcome. Events caused by the command record the same ID as their causation ID for traceability, but causation ID is not globally unique in the event store. If the same command arrives twice — from a retry, a network duplicate, or a process manager re-delivery — the command ledger returns the first result and no side effect is repeated. Nothing is recorded twice. Retrying is always safe.
6. Process managers coordinate without coupling Workflows that span more than one aggregate are handled by process managers — event-sourced finite state machines that listen for events and issue commands in response. They are the only components that know about more than one aggregate.

Process managers save their coordination state durably as events in their own event stream. On crash and restart they replay their events to recover their exact position in the workflow. They never restart from the beginning. They never duplicate work.
7. Domain events and integration events are different things Domain events are stored in Postgres in the event store. They are the source of truth. Integration events are published to RabbitMQ by the outbox publisher. They are the transport mechanism that notifies other parts of the system.

RabbitMQ provides at-least-once delivery. Consumers must be idempotent. The outbox pattern ensures no domain event is ever silently lost — even if RabbitMQ is unavailable, events accumulate in Postgres and are delivered when the broker recovers. Every consumer checks message or event IDs before acting. Processing the same event twice produces the same result as processing it once.
8. The URL structure reflects the domain model Each URL segment corresponds to a DDD aggregate. Action names follow the Stripe convention — POST to the aggregate root creates a new instance, POST to /{id}/{action} performs a named task on an existing one. The URL tells you the aggregate, the instance, and the exact task. No documentation is needed to understand what an endpoint does. The name is the documentation.

All endpoints use POST. All commands are idempotent. All requests carry a correlation ID. These are conventions — not suggestions.
9. Security is layered — each layer assumes the others can fail Cloudflare is the outer perimeter — WAF, DDoS, bot protection, rate limiting, with no open inbound ports on the registry server via Cloudflare Tunnel. Authentication via OIDC PKCE BFF ensures tokens never reach the browser — they live in Redis, encrypted by OpenBao Transit, keyed by a session ID in an HttpOnly cookie. Role enforcement happens in the handler. Business rule enforcement happens in the Decider. The command ledger enforces idempotency at the application boundary.

No single layer is sufficient. Each assumes the others can be bypassed. Compromise one layer and the others still hold.
10. Auditability is built into the data model Every event records who did it, when, with what outcome, and which command caused it (causation ID) and which original request triggered the whole chain (correlation ID). The full legal history of how any company came to exist is traceable across three aggregate event streams — from the first click of a button to the moment the company entered the Register.

This is not instrumentation added after the fact. It is the architecture. Event sourcing, CQRS, causation IDs, and correlation IDs are not engineering preferences. For a system on which the national economy depends, they are legal requirements.
Domain modelling 11. Ubiquitous language is not optional Every term in the code — every command name, event name, aggregate name, state name — must match the language lawyers and business analysts use exactly. When code drifts from the language, the domain model drifts from the law. RegistrationApplication is not ApplicationForm. RegisteredCompany is not Company. The names are precise because the domain is precise. 12. Terminal states are legal facts When an aggregate reaches a terminal state — Approved, Rejected, or Withdrawn on a RegistrationApplication; Submitted, Cancelled, or Expired on a Draft — that is not just a technical flag. It is a legal conclusion. The system enforces it by refusing all further commands. This is not a technical limitation — it is the law. 13. Draft the state machine before writing requirements Most requirements fall out of the state machine naturally. If you cannot draw the state machine, you do not understand the domain well enough to write code. The state machine is reviewable by a lawyer. The code is not.
Event sourcing 14. Never update or delete events Events are facts. Facts do not change. If something was recorded incorrectly, the correction is a new event — a compensating fact — not an edit to the original. The audit trail must be complete and tamper-evident. 15. Snapshots are an optimisation, not an architecture If replaying a long event stream becomes slow, snapshots can short-circuit the replay. But snapshots are a performance optimisation applied to a working system — not something to design around upfront. Start without them. 16. Projections are cheap. Events are precious. You can always add a new read model projection by replaying events. You cannot recover events you failed to record. When in doubt about whether to record something as an event, record it.
Distributed systems 17. Design for failure, not for success Every component — the database, the message broker, the identity broker, the address validation service — can fail at any time. The architecture assumes failure is normal. The outbox pattern, idempotent consumers, process manager state persistence, and at-least-once delivery are all responses to this assumption. 18. Eventual consistency is a feature, not a bug The examiner work queue read model will not reflect a new submission instantly — it will reflect it when the projector processes the integration event. For a registry processing company registrations this delay is measured in milliseconds and is completely acceptable. Designing for strict consistency everywhere would make the system fragile and slow. 19. Make implicit assumptions explicit Every assumption about ordering, timing, and consistency that is not enforced by the architecture will eventually be violated in production. The command ledger, the sequence number uniqueness constraint, and the process manager idempotency check all exist because the implicit assumption — "this will only happen once" — is not safe to leave implicit.
Operations 20. Observability is not optional Correlation IDs, causation IDs, structured logs, and metrics are not nice-to-haves. For a national registry, when something goes wrong at 2am, a developer must be able to trace exactly what happened and why using the data already in the system. If the data is not there, the answer is not there. 21. The read model is always suspect A read model can be stale, partially rebuilt, or temporarily inconsistent. Any feature that depends on a read model being perfectly up to date is fragile. Design the UI and the API to tolerate eventual consistency in read models. When you need the ground truth, go to the event store. 22. Configuration, not code Values that will change — the draft expiry period, the maximum number of active drafts per applicant, rate limits — must be configuration, not hardcoded constants. The law may change. Policy may change. The system must be able to change with it without a deployment.
