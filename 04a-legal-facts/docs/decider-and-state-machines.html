Deciders and State Machines

This document explains two concepts that appear throughout this codebase: state machines and the Functional Event Sourcing Decider pattern. If you are new to the project, read this before reading any aggregate code. Everything else will make more sense.

1. Why we have state machines

This is a national companies registry. Every action in the system has legal consequences. A registration application that is approved when it should have been rejected is not a bug — it is a legal liability. A draft that is submitted twice creates a duplicate application. A company that can be struck off after it has already been dissolved corrupts the legal record.

We prevent these outcomes not by writing defensive if-statements scattered through the codebase, but by encoding the rules in one place: the state machine. The state machine defines precisely which actions are permitted in which situations. If an action is not in the state machine, it cannot happen. Full stop.

The core principle Every aggregate in this system has a state machine. The state machine is the law made code. Before any business rule is checked, the state machine is consulted. If the requested action is not permitted in the current state, the request is rejected — no exceptions, no special cases, no workarounds.

2. What a state machine is

A state machine is a model of a system that can be in exactly one of a finite set of states at any time. It moves between states via transitions, which are triggered by commands. Each transition is only permitted when the system is in a specific state.

Here is the Draft aggregate's state machine in full:

Active →→ Submitted (SubmitDraft — validation must pass)
Active →→ Cancelled (CancelDraft — Applicant only)
Active →→ Expired (ExpireDraft — scheduler only)

All states except Active are terminal.

This tells us everything we need to know about what is allowed:

A draft in Active state can be submitted, cancelled, or expired. Nothing else. A draft in Submitted, Cancelled, or Expired state cannot do anything — these are terminal. Any command against a terminal draft is rejected before a single business rule is evaluated.

Terminal states A terminal state is one from which there are no outgoing transitions. Once reached, the aggregate is done. It will never change again. In this system, terminal states carry legal significance — a cancelled draft is a permanent legal record that no one may alter. The state machine enforces this by simply having no transitions out of Cancelled.

3. The Decider pattern

The Functional Event Sourcing Decider is a pattern described by Jérémie Chassaing. It gives a precise, functional structure to the business logic of an event-sourced aggregate.

The central idea is this: the aggregate's business logic is a pure function. Given the current state of the aggregate and a command, it returns a list of events. Nothing else. No database calls. No side effects. No network requests. The same inputs always produce the same outputs.

Why a pure function? A pure function is trivially testable — you call it with inputs and check the outputs. There is no mocking, no database setup, no test containers. Just data in, data out.

It is also the only correct approach for event sourcing. The business logic must not touch infrastructure. If it did, replaying events to rebuild state would trigger side effects. The Decider enforces this discipline structurally.

4. The three functions

Every Decider in this system implements exactly three functions:

initialState Returns the starting state for a brand new aggregate — one that has never received any events. 
;; Draft starts with no information
(defn initial-state []
  {:status  :new
   :name    nil
   :directors []
   :address nil})
evolve — (state, event) → state Takes the current state and one event. Returns the new state. This is how aggregate state is rebuilt from the event history. It must be a pure function — no side effects. 
(defn evolve [state event]
  (case (:event/type event)
    :DraftCreated     (assoc state :status :active
                                   :applicant-id (:applicant-id event))
    :DraftUpdated     (merge state (select-keys event [:name :directors :address]))
    :DraftSubmitted   (assoc state :status :submitted)
    :DraftCancelled   (assoc state :status :cancelled)
    :DraftExpired     (assoc state :status :expired)
    ;; unknown events are ignored — forward compatibility
    state))
decide — (state, command) → [events] or error Takes the current state and a command. Returns a list of events to record, or throws an error if the command is rejected. This is where the state machine and business rules live. 
(defn decide [state command]
  (case (:command/type command)

    :SubmitDraft
    (do
      ;; 1. State machine check — ALWAYS FIRST
      (when-not (= :active (:status state))
        (throw (ex-info "Draft cannot be submitted"
                        {:error :invalid-state
                         :current-state (:status state)})))
      ;; 2. Business rules — only if state permits
      (when-not (:name state)
        (throw (ex-info "Company name required" {:error :BR-SB-001})))
      (when (< (count (:directors state)) 2)
        (throw (ex-info "At least two directors required" {:error :BR-SB-002})))
      ;; ... more rules ...
      ;; 3. Return events
      [{:event/type   :DraftSubmitted
        :draft-id     (:draft-id state)
        :applicant-id (:applicant-id state)}])

    :CancelDraft
    (do
      (when-not (= :active (:status state))
        (throw (ex-info "Draft cannot be cancelled"
                        {:error :invalid-state})))
      [{:event/type   :DraftCancelled
        :draft-id     (:draft-id state)
        :applicant-id (:applicant-id state)}])

    ;; Unknown command
    (throw (ex-info "Unknown command" {:command (:command/type command)}))))
The two-layer check in decide — this order is mandatory The state machine check always comes first. If the current state does not permit this command, we stop immediately. We do not run business rules. We do not look at the data. We return an error.

Only if the state machine permits the command do we run business rules. This is not a convention — it is a requirement. Running business rules on a command that the state machine has already rejected wastes computation and can produce misleading error messages.

5. The runner — how it all connects

The Decider knows nothing about the database, the network, or the HTTP request. Something else must load events, call the Decider, and save the results. That something is the runner.

The runner follows the same steps for every command, for every aggregate: 

1. Load all past events for this aggregate from the event store
   SELECT * FROM draft_events
   WHERE  aggregate_id = ?
   ORDER  BY sequence_number ASC

2. Fold events through evolve to rebuild current state
   state = reduce evolve (initial-state) events

3. Enrich the command with a command ID and correlation ID
   command = assoc command :command/id (new-uuid)

4. Call decide(state, command) → [new-events] or error

5. Save new-events to the event store
   (optimistic concurrency: reject if sequence number conflict)
   (outbox: mark published=false for delivery to RabbitMQ)

6. Return success to the caller
Fold — rebuilding state from history Folding is a functional programming concept. Starting from initial-state, each event is applied to the current state using evolve to produce the next state. After all events are folded, the result is the exact current state of the aggregate. 
;; Pseudocode — the runner does this
(defn load-state [aggregate-id]
  (let [events (load-events aggregate-id)]
    (reduce evolve (initial-state) events)))
There is no state column in the database. There is no snapshot of "current state." The state is always computed from the event history. This is event sourcing. The event store is the ground truth.
Optimistic Concurrency Control When saving events, the runner checks that the sequence number of the last event it read is still the last event in the store. If another process wrote events in the meantime, the sequence number will have advanced and the write is rejected. The command can be retried. This prevents two concurrent operations from corrupting the aggregate's state without using database locks.

6. Full example: the Draft aggregate

The Draft is the simplest aggregate in the system. Here is its complete Decider, matching exactly what the business rules specify. 

(ns registry.draft.decider)

;; ── State ─────────────────────────────────────────────────────────────────

(defn initial-state []
  {:status       :new        ;; before DraftCreated is processed
   :draft-id     nil
   :applicant-id nil
   :name         nil
   :directors    []
   :address      nil})

;; ── Evolve ────────────────────────────────────────────────────────────────

(defn evolve [state {:event/keys [type] :as event}]
  (case type
    :DraftCreated    (assoc state
                            :status       :active
                            :draft-id     (:draft-id event)
                            :applicant-id (:applicant-id event))
    :DraftUpdated    (merge state
                            (select-keys event [:name :directors :address]))
    :DraftSubmitted  (assoc state :status :submitted)
    :DraftCancelled  (assoc state :status :cancelled)
    :DraftExpired    (assoc state :status :expired)
    state))  ;; unknown events → no change

;; ── Decide ────────────────────────────────────────────────────────────────

(defmulti decide (fn [_state command] (:command/type command)))

(defmethod decide :CreateDraft
  [state {:keys [draft-id applicant-id]}]
  ;; State machine: only :new accepts :CreateDraft
  (when-not (= :new (:status state))
    (throw (ex-info "Draft already exists" {:error :invalid-state})))
  [{:event/type   :DraftCreated
    :draft-id     draft-id
    :applicant-id applicant-id}])

(defmethod decide :UpdateDraft
  [state {:keys [name directors address]}]
  ;; BR-DR-006/007/008: only active drafts may be updated
  (when-not (= :active (:status state))
    (throw (ex-info "Only active drafts may be updated"
                    {:error :invalid-state :current (:status state)})))
  [{:event/type :DraftUpdated
    :draft-id   (:draft-id state)
    :name       name
    :directors  directors
    :address    address}])

(defmethod decide :CancelDraft
  [state {:keys [requesting-applicant-id]}]
  ;; BR-DR-005: active only
  (when-not (= :active (:status state))
    (throw (ex-info "Only active drafts may be cancelled"
                    {:error :invalid-state})))
  ;; BR-DR-011: own drafts only
  (when-not (= (:applicant-id state) requesting-applicant-id)
    (throw (ex-info "Cannot cancel another applicant's draft"
                    {:error :forbidden})))
  [{:event/type   :DraftCancelled
    :draft-id     (:draft-id state)
    :applicant-id (:applicant-id state)}])

(defmethod decide :ExpireDraft
  [state _command]
  ;; BR-DR-004: only active drafts expire
  (when-not (= :active (:status state))
    (throw (ex-info "Only active drafts may be expired"
                    {:error :invalid-state})))
  [{:event/type   :DraftExpired
    :draft-id     (:draft-id state)
    :applicant-id (:applicant-id state)}])

(defmethod decide :SubmitDraft
  [state _command]
  ;; 1. State machine
  (when-not (= :active (:status state))
    (throw (ex-info "Only active drafts may be submitted"
                    {:error :invalid-state})))
  ;; 2. Business rules (BR-SB-001 to BR-SB-006)
  (let [errors
        (cond-> []
          (not (:name state))
          (conj {:error :BR-SB-001 :message "Company name required"})

          (< (count (:directors state)) 2)
          (conj {:error :BR-SB-002 :message "At least two directors required"})

          (not (apply distinct? (map :verification-id (:directors state))))
          (conj {:error :BR-SB-003 :message "Directors must be distinct persons"})

          (not (:address state))
          (conj {:error :BR-SB-004 :message "Registered office address required"})

          (not (every? :natural-person? (:directors state)))
          (conj {:error :BR-SB-005 :message "All directors must be natural persons"})

          (not (every? :verified? (:directors state)))
          (conj {:error :BR-SB-006 :message "All directors must be identity-verified"}))]
    (when (seq errors)
      (throw (ex-info "Submission validation failed" {:errors errors}))))
  ;; 3. Return event
  [{:event/type   :DraftSubmitted
    :draft-id     (:draft-id state)
    :applicant-id (:applicant-id state)
    :name         (:name state)
    :directors    (:directors state)
    :address      (:address state)}])

;; ── Terminal? ─────────────────────────────────────────────────────────────

(defn terminal? [state]
  (contains? #{:submitted :cancelled :expired} (:status state)))

7. Full example: the Registration Application aggregate

The Registration Application has a more complex state machine with four non-terminal states. The structure is identical to the Draft — the pattern scales cleanly.

Submitted →→ UnderExamination (BeginExamination)
UnderExamination →→ AwaitingRequisitionResponse (RaiseRequisition)
AwaitingRequisitionResponse →→ UnderExamination (RequisitionAnswered)
UnderExamination →→ ReadyForDecision (ReferToRegistrar)
ReadyForDecision →→ Approved (ApproveApplication)
ReadyForDecision →→ Rejected (RejectApplication)
Submitted|UnderExamination|
AwaitingRequisitionResponse|
ReadyForDecision →→ Withdrawn (WithdrawApplication)

Terminal: Approved, Rejected, Withdrawn

Notice that all four non-terminal states permit WithdrawApplication. This reflects §11 of the Act — withdrawal is permitted at any time before a decision. The state machine enforces the law. 

(defn evolve [state {:event/keys [type] :as event}]
  (case type
    :RegistrationApplicationCreated
    (assoc state
           :status         :submitted
           :application-id (:application-id event)
           :source-draft-id (:source-draft-id event)
           :name           (:name event)
           :directors      (:directors event)
           :address        (:address event)
           :applicant-id   (:applicant-id event))

    :ExaminationStarted
    (assoc state
           :status      :under-examination
           :examiner-id (:examiner-id event))

    :RequisitionRaised
    (-> state
        (assoc :status :awaiting-requisition-response)
        (update :open-requisitions conj (:requisition-id event)))

    :RequisitionAnswered
    (let [remaining (remove #{(:requisition-id event)}
                            (:open-requisitions state))]
      (cond-> state
        true           (update :open-requisitions disj (:requisition-id event))
        (empty? remaining) (assoc :status :under-examination)))

    :ReferredToRegistrar
    (assoc state :status :ready-for-decision)

    :RegistrationApplicationApproved
    (assoc state :status :approved :registrar-id (:registrar-id event))

    :RegistrationApplicationRejected
    (assoc state :status :rejected :rejection-reason (:reason event))

    :RegistrationApplicationWithdrawn
    (assoc state :status :withdrawn)

    state))

;; The decide function follows exactly the same two-layer pattern:
;; 1. State machine check
;; 2. Business rules (only if state permits)
;; 3. Return events
The Decider has no knowledge of who is calling it Role enforcement — checking that only an Examiner can begin examination, or only the Registrar can approve — happens in the handler, before the command reaches the runner. The Decider receives a fully validated command. It trusts that the caller has already been authorised.

This is the Separation of Concerns principle applied to security. Security policy lives in the handler. Domain logic lives in the Decider. They never mix.

8. How business rules map to Decider code

Every business rule in the Business Rules document has a code location. The mapping is direct and intentional.

State machine checks → top of each decide method Rules like BR-DR-006 ("a submitted draft may not be updated") map to the state machine check at the top of the :UpdateDraft method. If the status is not :active, throw immediately.
Business rules → validation block after state check Rules like BR-SB-001 ("draft must state a company name") map to the validation block inside the :SubmitDraft method. They run only after the state machine has permitted the command.
Injected functions for external checks Rules like BR-AP-007 (name uniqueness against the Register) and BR-AP-009 (address validation) cannot be checked by the Decider alone — they require authoritative external facts. These are passed into the command as functions: 
;; The handler injects these — the Decider calls them
{:command/type      :ApproveApplication
 :name-available-fn (fn [name] (register-write-side/name-available? store name))
 :address-valid-fn (fn [addr] (address-service/valid? addr))}
The Decider calls them and uses the result. It never calls the database or the address service directly. The Decider remains pure — the injected functions are its only window onto the outside world, and the handler is responsible for providing correct implementations. The name check must use authoritative write-side Register facts, not a stale read model. In this contract it is a non-reserving check: name locking at draft or submission time is parked for future legal and product design.
The "Previously" column is gone — use BR IDs in code comments Every business rule check in the Decider should reference its BR ID in a comment. When a developer reads the code, they can look up the rule in the Business Rules document and find the Act section it derives from. This is the traceability chain: Act → Business Rules → Code. 
;; BR-SB-001: company name must be stated (§8)
(when-not (:name state)
  (throw (ex-info "Company name required" {:error :BR-SB-001})))

9. Testing a Decider

Because the Decider is a pure function, testing is simple. No mocks. No database. No test containers. Just data. 

(deftest submit-draft-test

  (testing "happy path — all rules pass"
    (let [state  (build-valid-active-draft)
          command {:command/type :SubmitDraft}
          events (decide state command)]
      (is (= 1 (count events)))
      (is (= :DraftSubmitted (:event/type (first events))))))

  (testing "BR-DR-006 — cannot submit a cancelled draft"
    (let [state   (assoc (build-draft) :status :cancelled)
          command {:command/type :SubmitDraft}]
      (is (thrown-with-msg?
            clojure.lang.ExceptionInfo
            #"invalid-state"
            (decide state command)))))

  (testing "BR-SB-001 — company name required"
    (let [state   (assoc (build-valid-active-draft) :name nil)
          command {:command/type :SubmitDraft}
          ex      (try (decide state command)
                       (catch Exception e e))]
      (is (= :BR-SB-001 (:error (ex-data ex))))))

  (testing "state machine check runs before business rules"
    ;; A cancelled draft with missing name should fail on state machine,
    ;; not on BR-SB-001
    (let [state   (assoc (build-draft) :status :cancelled :name nil)
          command {:command/type :SubmitDraft}
          ex      (try (decide state command) (catch Exception e e))]
      (is (= :invalid-state (:error (ex-data ex)))))))

The last test is important. It verifies that the state machine check runs before business rules. A cancelled draft should always fail on state, not on data — regardless of what data it contains.

10. Common mistakes

Mistake 1 — putting business rules before the state machine check 
;; WRONG — business rule checked before state machine
(defmethod decide :SubmitDraft [state command]
  (when-not (:name state)            ;; ← business rule runs first
    (throw ...))
  (when-not (= :active (:status state))  ;; ← state machine check too late
    (throw ...))
  ...)
This means a cancelled draft with a missing name will produce the wrong error. Always check the state machine first.
Mistake 2 — side effects inside the Decider 
;; WRONG — the Decider calls the database directly
(defmethod decide :ApproveApplication [state command]
  (when-not (= :ready-for-decision (:status state)) (throw ...))
  (let [exists? (db/name-exists? (:db command) (:name state))]  ;; ← NO
    ...)
Pass a function into the command instead. The Decider calls the function, not the database. This keeps the Decider pure and testable without a database.
Mistake 3 — storing state outside the event stream 
;; WRONG — saving current state to a separate table
(defmethod decide :SubmitDraft [state command]
  ...
  (db/update-draft-status! (:draft-id state) :submitted)  ;; ← NO
  [{:event/type :DraftSubmitted ...}])
State is rebuilt by replaying events. There is no state table. Writing to one creates a second source of truth that will drift from the event stream.
Mistake 4 — ignoring unknown events in evolve 
;; WRONG — throwing on unknown events
(defn evolve [state event]
  (case (:event/type event)
    :DraftCreated   ...
    :DraftUpdated   ...
    (throw (ex-info "Unknown event" {:type (:event/type event)})))  ;; ← NO
The correct behaviour is to return the state unchanged for unknown events. This is forward compatibility — a future version of the system may emit new event types that an older consumer should simply ignore.
Mistake 5 — bypassing the Decider for scheduled jobs 
;; WRONG — scheduler writes events directly
(defn expire-drafts! [db]
  (doseq [draft-id (find-expired-drafts db)]
    (db/insert-event! db {:event/type :DraftExpired    ;; ← bypasses Decider
                          :draft-id draft-id})))
The expiry scheduler must issue an ExpireDraft command to the runner. The runner calls the Decider. The Decider checks the state machine and emits the event. Bypassing this path bypasses the audit trail, the state machine, and any business logic — all of which may be legally significant in this system.

11. All aggregates in this system

There are three domain aggregates and one process manager family in this system.

Draft The Applicant's private workspace. Enforces the 5-draft limit, expiry, and all submission validation rules. Non-terminal states Active Terminal states Submitted, Cancelled, Expired Decider registry.draft.decider
Registration Application The formal lodgement under examination. Manages the examination workflow including requisitions, referral, and the Registrar's decision. Non-terminal states Submitted, UnderExamination, AwaitingRequisitionResponse, ReadyForDecision Terminal states Approved, Rejected, Withdrawn Decider registry.registration-application.decider
Registered Company The legal entity. Records director changes, address changes, annual returns, and the end of the company's life (strike-off or dissolution). Non-terminal states Registered Terminal states StruckOff, Dissolved Decider registry.registered-company.decider
Process Managers — coordination, not domain Process managers are event-sourced finite state machines, but they are not domain aggregates. They coordinate work across aggregates by listening for events and issuing commands. They use the same Decider structure — evolve, decide, initial-state — but their state represents coordination progress, not domain facts.

Current process managers:

DraftSubmissionProcessManager — listens for DraftSubmitted, creates the RegistrationApplication
ApprovalProcessManager — listens for RegistrationApplicationApproved, creates the RegisteredCompany

Process managers must persist their coordination state durably. If they crash between receiving an event and issuing the resulting command, they must recover to exactly the right position. They never restart from the beginning. This is why they are event-sourced.
